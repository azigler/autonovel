# Implementation Orchestrator Workflow

You are the orchestrating agent managing implementation phases. This skill is for
planning and dispatching implementation work via beads and subagents. Individual
implementers use `/beads` + `/commit`; you use this skill to coordinate them.

All agents (orchestrator and implementers) should know `/beads` and `/commit`.
The orchestrator should also know `/branch` for branch management and releases.

## Expected Flow: Review -> Spec Updates -> Tests -> Implementation

```
/review all                    <- Walk through all open questions and conflicts
    |
Implementation Readiness Report <- Generated by /review when blockers are cleared
    |
Apply spec updates              <- Resolve "Spec update needed" notes from review
    |
/branch: create branch          <- Create branch from main
    |
/impl plan                      <- Plan the scope of work for this branch
    |
/test (write tests first)       <- Test-first for the relevant specs (on branch)
    |
/impl dispatch                  <- Build until tests pass (on branch)
    |
/branch: merge + tag + release  <- Merge branch to main, tag, release
    |
Repeat for next scope of work
```

**Do not start implementation until the readiness report confirms the scope is READY.**

## Step 1: Load Context

### Required Reading (every planning session)
1. **Implementation Readiness Report** -- generated by `/review` at end of review.
   Contains readiness status, spec updates needed, and recommended order.
2. **Project design decisions** -- locked decisions, resolved open questions
3. **`/test` skill** -- test creation workflow, naming conventions

### Blockers Check

Before planning any scope of work, verify:

1. **Open question blockers**: All open questions affecting this scope must be DECIDED.
2. **Conflict blockers**: All cross-spec conflicts involving this scope must be RESOLVED.
3. **Dependency blockers**: All cross-spec dependencies must be CONFIRMED.
4. **Spec updates**: All "Spec update needed" notes affecting this scope must be applied.

If any blocker is unresolved, invoke `/review` to resolve it before proceeding.

## Step 2: Create Branch

Before planning or dispatching work, create a branch per `/branch`:

```bash
git checkout main
git checkout -b v0.N/descriptive-name
git push -u origin v0.N/descriptive-name
```

All worktree agents for this scope merge into the branch, not `main`.
See `/branch` for the full lifecycle.

## Step 3: Plan the Scope

For a given scope of work, produce a plan with these sections:

### 3.1 Scope

```
Scope: [Name]
Branch: v0.N/descriptive-name
Specs: [list of spec numbers feeding into this scope]
Depends on: [prior work that must be complete]
Open question status: [all resolved / N unresolved -- list them]
```

### 3.2 Test-First Beads

Tests are written before implementation. Create beads in this order:

```bash
br create -p 2 "tests: spec NN [subsystem]"
br update <id> --description "$(cat <<'EOF'
## Context
Tests covering Spec NN for [subsystem].

## Task
Follow /test skill. Convert spec test cases to tests.
Add 5+ edge cases. Mark unimplemented deps appropriately.

## Acceptance Criteria
- [ ] Every Spec NN Section 5 test case has a test
- [ ] 5+ additional edge/error tests
- [ ] Tests compile (placeholders for missing deps)
- [ ] Test file in correct location per project convention
EOF
)"
```

### 3.3 Implementation Beads

After test beads, create implementation beads:

```bash
br create -p 2 "impl: [module name]"
br update <id> --description "$(cat <<'EOF'
## Context
Implement [subsystem] per Spec NN. Tests in [test file path].

## Task
Make all non-skipped tests pass. Follow spec Section 4 (formal spec)
and Section 6 (implementation notes).

## Acceptance Criteria
- [ ] All tests pass
- [ ] Linting clean
- [ ] Doc comments on public API
- [ ] No panicking/crashing code in non-test paths
EOF
)"
```

### 3.4 Integration Beads

For scopes that combine subsystems:

```bash
br create -p 2 "integration: [scope]"
br update <id> --description "$(cat <<'EOF'
## Context
Integration tests combining prior subsystems.

## Task
Write and pass integration tests.

## Acceptance Criteria
- [ ] End-to-end test covering [specific scenario]
- [ ] All unit tests still pass
EOF
)"
```

## Step 4: Dispatch Implementation

For each bead, define the subagent prompt:

### Subagent Prompt Template (Implementation)

```
Your bead is `<bead-id>`. Include `Bead: <bead-id>` in your commit trailer.
Merge target: `v0.N/descriptive-name` (NOT main). See /branch skill.

## CRITICAL: No Nested Agents
You are a subagent. Do NOT use the Agent tool to spawn further subagents.
Do all work directly using Read, Write, Edit, Bash, Grep, and Glob.

## Task
Implementation for [Name].

## Scope
- Branch: `v0.N/descriptive-name`
- Module: `[module name]`
- Spec: `specs/NN-name.md` (read Sections 4, 5, 6)
- Test file: `[test file path]`
- Depends on: [list of modules that must exist]

## CRITICAL RULES
1. Do NOT use the Agent tool. You are a leaf agent -- do all work directly.
2. Do NOT create or modify test files (`tests/`, `test_*.py`). The tests
   define the contract -- adapt your implementation to match the tests, not the
   other way around.
3. Read the test file FIRST to understand what it imports and expects.
4. Your module's public API (exports, function signatures, type names) must
   match what the tests import.

## Key Decisions
[List resolved open questions relevant to this scope, with their answers]

## Cross-Cutting Requirements (MANDATORY)
[Copy the full quality checklist into every prompt -- see Step 7]

## Pre-Commit Verification (MANDATORY)
Before committing, run the project's quality checks and fix any failures:
- All tests pass
- Linting clean
- No unhandled errors in non-test code
- Doc comments on public API
```

### Subagent Prompt Template (Tests)

```
Your bead is `<bead-id>`. Include `Bead: <bead-id>` in your commit trailer.
Merge target: `v0.N/descriptive-name` (NOT main). See /branch skill.

## CRITICAL: No Nested Agents
You are a subagent. Do NOT use the Agent tool to spawn further subagents.
Do all work directly using Read, Write, Edit, Bash, Grep, and Glob.

## Task
Test-writing for [Name].

## Scope
- Branch: `v0.N/descriptive-name`
- Module: `[module name]`
- Spec: `specs/NN-name.md` (read Section 5 for test cases)
- Output: `[test file path]`
- Depends on: [list of modules from prior work]

## CRITICAL RULES
1. Do NOT use the Agent tool. You are a leaf agent -- do all work directly.
2. Write ONLY test files. Do NOT create or modify implementation source files.
3. Import from real module paths (not inline stubs). Tests will fail until impl
   lands -- that is correct TDD behavior.
4. Follow /test skill naming conventions (TEST/EDGE/ERROR prefixes).
5. Every Section 5 test case gets a corresponding test.
6. Add at least 5 edge/error tests beyond the spec.
```

### Dependency Ordering

Test beads MUST complete before their corresponding impl beads. **Critically,
test worktrees must be merged and cleaned up before dispatching impl agents.**
Otherwise impl agents branch from a stale base and miss the test files.

```
1. Dispatch test bead(s) -- can parallelize across specs in same scope
2. Wait for test beads to complete
3. Verify test agents wrote ONLY test files (no implementation source changes)
4. **Merge all test worktrees into the version branch**
5. **Clean up all test worktrees and branches immediately**
6. **Push the version branch**
7. Dispatch impl bead(s) -- can parallelize across independent modules
8. Wait for impl beads to complete, merge each
9. Run Quality Gate (Step 4.5) on the branch
10. Dispatch integration bead (if applicable)
11. Mark scope complete
```

**Why steps 4-6 are critical:** Worktree subagents branch from the current HEAD
at the time they are created. If test worktrees are not merged first, impl agents
will not have the test files and may rewrite tests to match their implementation
rather than the spec. This defeats the purpose of TDD.

### Step 4.5: Quality Gate (MANDATORY before merging branch to main)

After all impl beads are merged into the branch, run project-appropriate
verification before proceeding to `/branch` merge + tag:

```bash
# Run all tests
# Run linter
# Check for unhandled errors in non-test code
# Verify doc comments on public API
```

If any check fails, create a fix bead and dispatch a cleanup agent before merging.
Do NOT merge a branch to main with known quality gaps.

## Step 6: Cross-Cutting Requirements

Include this checklist in EVERY implementation subagent prompt. Adapt the specific
tooling to your project's language and build system:

### Mandatory for All Implementations

- **Do not modify test files**: Tests define the contract. Adapt your
  implementation to match what the tests import and expect. If a test seems
  wrong, flag it -- do not rewrite it.
- **Test isolation**: All test-only code clearly separated from production code.
- **No panicking in production code**: Use proper error handling.
- **Lint clean**: All linter checks must pass. No suppressions without justifying comments.
- **Doc comments on public API**: Every public function, type, and module
  gets a doc comment.
- **Error types**: Define proper error types per the project's convention.
  Never use string errors.
- **Consistent naming**: Follow language conventions.
